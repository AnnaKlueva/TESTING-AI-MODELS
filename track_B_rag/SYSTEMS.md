# Демо-системи: опис і вимоги до тестування

Обери **одну** систему. Нижче для кожної — **опис** (що це й через що працює) та **вимоги до тестування**
(що QA має перевірити). Вимоги кажуть **ЩО** перевірити; **ЯК** (стратегія, метрики, пороги, набір кейсів) —
твоє оцінюване рішення (див. `RUBRIC.md`, `CLAUDE.md`). Дефекти в системах не позначені — це grey-box.

| Трек | Складність запуску | Головні ризики | Найбільше зачіпає |
|---|---|---|---|
| A. Chatbot | середня (локальна LLM) | пам'ять, безпека, відмови | Блоки 3, 5 |
| B. RAG | середня (ембединги+LLM) | пошук, галюцинації, EN/UA | Блоки 4, 3 |
| C. ML | легка (sklearn) | зрізи, дрейф, регресія | Блок 2 |
| D. Agent | **найлегша (без GPU/ключів)** | маршрутизація, інструменти, tool-poisoning | практикум «Agentic QA» (зан. 16), Блок 5 |

## Трек A — Chatbot (`chatbot_sut.py`)

**Опис.** Асистент підтримки Acme Cloud на базі локальної LLM. Приймає повідомлення користувача,
тримає коротку історію діалогу і повертає текст. Інтерфейс: `ChatbotSUT().chat(msg)`, `.reset()`.
Має системний промпт із політикою (не розкривати внутрішні дані, відмовляти в певних запитах).

**Вимоги до тестування (що перевірити).**
1. **Пам'ять діалогу** — чи система пам'ятає раніше повідомлене в межах сесії.
2. **Дотримання інструкцій/політики** — чи відповіді відповідають системному промпту.
3. **Безпечні відмови** — коректна відмова на заборонені запити в **будь-якому** формулюванні.
4. **Безпека** — стійкість до prompt-injection; невитік внутрішніх даних/токенів.
5. **Стабільність** — повторюваність за фіксованих параметрів (недетермінізм).

## Трек B — RAG (`rag_sut.py`)

**Опис.** Питально-відповідна система над базою знань (EN + UA): запит → ембединг → пошук у Chroma →
контекст → локальна LLM → відповідь. Інтерфейс: `RagSUT().ask(q)` → `{answer, sources}`, `.retrieve(q)`.

**Вимоги до тестування (що перевірити).**
1. **Якість пошуку** — чи знайдено релевантні джерела для запиту.
2. **Заземленість (faithfulness)** — чи відповідь спирається на знайдений контекст, а не вигадана.
3. **Коректність** — чи відповідь відповідає актуальній інформації в базі.
4. **Безпечна відмова** — «не знаю», коли відповіді немає в базі (без фабрикації).
5. **Крос-мовність** — узгодженість EN та UA для еквівалентних запитів.

## Трек C — ML-класифікатор (`sut_artifact.joblib` + `production_data.csv`)

**Опис.** Навчений табличний класифікатор (sklearn-пайплайн) + мічений production-набір.
Інтерфейс: завантаження артефакту, `predict` / `predict_proba` по рядках `production_data.csv`.

**Вимоги до тестування (що перевірити).**
1. **Якість по зрізах** — чи не провалюється модель на окремих сегментах (не лише загальна метрика).
2. **Пастка точності** — чи адекватні precision/recall при дисбалансі класів.
3. **Дрейф даних** — чи розподіл production відрізняється від тренувального.
4. **Надійність ознак** — чи немає ознаки, що «лікає» і не генералізується.
5. **Регресія** — порівняння з baseline (де доречно).

## Трек D — Agent (`agent_sut.py`)

**Опис.** Агентна система (як у практикумі «Agentic QA», зан. 16): orchestrator визначає намір → домен-агент викликає
інструменти (MCP-моки) → формується відповідь. Інтерфейс: `AgentSUT().handle(prompt)` →
`{output, selected_agent, tool_calls}`. Детермінований, без GPU/ключів (найлегший старт).

**Вимоги до тестування (що перевірити).**
1. **Маршрутизація наміру** — чи обрано правильного домен-агента під запит.
2. **Вибір інструментів (tool call accuracy)** — правильний інструмент і правильні аргументи.
3. **Траєкторія** — відсутність петель і надлишкових викликів.
4. **Виконання задачі** — чи фактично досягнуто мету запиту.
5. **Tool-poisoning / безпека** — агент не виконує інструкцій із виходу інструмента й не витікає токени.

---

## Що ти вирішуєш сам (оцінюється)
Для будь-якого треку **ти** обираєш і обґрунтовуєш:
- тестову стратегію й підхід (типи перевірок, пріоритизацію за ризиком);
- **метрики та їхні пороги** (чому саме ці під ці ризики);
- **дизайн кейсів** (happy / edge / negative / adversarial, ≥30);
- **аналіз результатів** (дефекти, severity, root cause, висновки).

Вимоги вище — це **що** перевірити (test basis). Твоя робота — перетворити їх на конкретну стратегію,
метрики й датасет, а потім проаналізувати, що показали прогони.
