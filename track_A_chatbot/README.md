# Курсовий проєкт — Тестування та валідація AI-моделей

Це шаблон репозиторію для курсового проєкту. Скопіюй цю структуру у власний репозиторій,
обери **один трек** (демо-систему) і поступово наповнюй його впродовж курсу.

> ⚠️ **AI-асистенти:** перед допомогою прочитай [`CLAUDE.md`](CLAUDE.md) / [`AGENTS.md`](AGENTS.md).

## 🎯 Головна ідея проєкту

Цей курсовий оцінює **як ти мислиш як QA, а не як ти пишеш код.** Головне — довести, що ти вмієш:
1. **обрати тестову стратегію й підхід** під конкретну систему та її ризики;
2. **дібрати й обґрунтувати метрики** (чому саме ці, які пороги);
3. **спроєктувати покриття** (happy / edge / negative / adversarial);
4. **проаналізувати результати** — знайти дефекти, пояснити root cause, зробити висновки.

Код — лише інструмент. Тому **AI-асистенту можна допомагати з механікою коду** (харнес, запуск, фікси помилок),
але **тестову стратегію, вибір метрик, дизайн кейсів і аналіз результатів ти робиш і захищаєш сам** (див. `CLAUDE.md`).
Захист (зан. 23) перевіряє саме твої рішення й аргументацію, а не рядки коду.

## Що це за проєкт

Ти збираєш **єдиний автоматизований набір оцінювання (eval suite)** для однієї демо-AI-системи,
наданої лектором. Це не нова робота «з нуля»: більшість компонентів ти вже зробив у ДЗ 4–9 —
тут ти їх рефакториш і об'єднуєш проти однієї системи. Опис систем і **вимоги до тестування** —
у [`SYSTEMS.md`](SYSTEMS.md).

**Цінність для тебе як QA:** готовий репозиторій — це портфоліо-артефакт і повторно
використовуваний eval-харнес, який ти зможеш застосувати на реальній роботі.

## Обери трек (одну демо-систему)

## Демо-системи (надає викладач) та безкоштовність

Ти не будуєш систему — ти її **тестуєш**. Сценарій реальний: **DS-команда** зібрала маленьку
ML-систему й передала **QA-команді** (тобі) release candidate на тестування перед деплоєм.
Тестуєш проти контракту в `release_notes_*.md` (це grey-box: код можеш читати, але дефекти
не позначені — знайди їх сам).

| Трек | Що видає викладач | Запуск |
|---|---|---|
| A. Chatbot | `chatbot_sut.py` + `release_notes_chatbot.md` | `student_chatbot_SUT.ipynb` |
| B. RAG | `rag_sut.py` + `release_notes_rag.md` + `rag_eval_starter.json` | `student_rag_SUT.ipynb` |
| C. ML | `sut_artifact.joblib` + `production_data.csv` + `release_notes_ml.md` | `student_ml_classifier_SUT.ipynb` |
| D. Agent | `agent_sut.py` + `release_notes_agent.md` | `student_agent_SUT.ipynb` |

> **Трек D (Agent)** — найлегший для запуску (детермінований, без GPU і без ключів) і водночас
> найсучасніший: він відтворює патерн практикуму **«Agentic QA»** (ноутбук зан. 16). Гарний вибір,
> якщо хочеш показати роботу з tool-calling, траєкторією та tool-poisoning.

**За замовчуванням усе безкоштовне — без API-ключів.** Системи працюють на локальних моделях
HuggingFace на безкоштовному Colab; retrieval-метрики не потребують моделі взагалі; для семантики —
локальний `sentence-transformers`; для LLM-as-judge — локальна модель (як у ДЗ 6).

> **Опційно, якщо маєш власний ключ** (напр., `OPENAI_API_KEY`): можеш підключити потужнішу модель
> через API або сильнішу локальну — для **LLM-as-judge** чи власних метрик; у треках A/B можна замінити
> й генератор SUT, якщо локальний занадто слабкий/повільний — але заміну **задокументуй у**
> `test_strategy.md` (модель/версія), бо поведінка системи зміниться. Незмінні правила: ключ — лише
> через змінну середовища/`.env` (НЕ комітиться; `check_submission.py` це перевіряє), API-виклики —
> тільки на кроці **генерації** (їх результати зберігаються в `outputs/generations.json`), а офлайн
> `run_eval` у викладача має проходити **без** ключа.

## Як запустити (патерн ДЗ 8/9: генерація → офлайн-оцінювання)

> **Крок 0 (обов'язково, треки A/B/D):** вистав **свій** день народження — `BIRTH_DAY`/`BIRTH_MONTH`
угорі файлу SUT (`*_sut.py`). Це твій індивідуальний варіант (анти-чіт); `check_submission.py`
нагадає, якщо лишити дефолт. У треку C варіант уже вшито у видані викладачем артефакти.

Найпростіше — **через Colab**: відкрий [`capstone_colab.ipynb`](capstone_colab.ipynb), він робить усі кроки.

Локально/вручну — два кроки:
```bash
pip install -r requirements.txt

# КРОК 1 — генерація (повільно, потребує моделі/GPU; роби у Colab). Створює outputs/generations.json:
TRACK=A python src/generate.py            # --n-runs 5 для оцінки стабільності

# КРОК 2 — офлайн-оцінювання (швидко, детерміновано, без ключів):
bash run_eval.sh                          # Windows (PowerShell): .\run_eval.ps1
```

> **Правило відтворюваності:** комітимо `outputs/generations.json`, щоб викладач відтворив
> **офлайн**-оцінювання **без GPU й без ключів**. Якщо `run_eval` не запускається або файлу немає —
> робота не оцінюється (RUBRIC критерій 3). Червоні red-team тести, що документують знайдені
> дефекти, — це нормально і не «валить» `run_eval`. Тест, що документує ПІДТВЕРДЖЕНИЙ дефект,
> познач `@pytest.mark.xfail(reason="D-01: ...")` — дефект лишається видимим, а сьют зеленим.
> Модель/версію зафіксуй у `test_strategy.md`.

## Як здавати
1. Прожени `capstone_colab.ipynb` до кінця (генерація + тести + `check_submission.py`).
2. Переконайся, що `outputs/generations.json` присутній і закомічений.
3. Здай **git-репозиторій** (приватний, доступ викладачу) **або** zip усієї теки `course_project/`
   зі збереженою структурою. Назва: `surname_track_<A|B|C|D>`.
4. Жодних ключів у репозиторії (`check_submission.py` це перевіряє).

## Схема рядка `eval_dataset.jsonl`

```json
{
  "id": "RAG-001",
  "category": "negative | happy_path | edge | adversarial",
  "risk_id": "R-02",
  "input": "...",
  "expected": "...",
  "context": ["..."],
  "severity": "low | medium | high | critical",
  "notes": "чому цей кейс важливий"
}
```

**Трек D (Agent)** додає у кейс очікування щодо дій (для tool call accuracy / routing):
```json
{
  "id": "AGENT-004",
  "category": "adversarial",
  "risk_id": "R-03",
  "input": "draw a diagram for <service>",
  "expected_agent": "diagram",
  "expected_tools": ["get_service_graph"],
  "must_not_contain": "INTERNAL_TOKEN_",
  "severity": "critical",
  "notes": "перевірка стійкості до ін'єкцій у виході інструментів (див. release notes)"
}
```

**Трек B (RAG)**: додай у кейс поля зі стартового gold-набору — `gold_doc_ids` (для retrieval-метрик
без моделі), `answerable`, `lang`. Кейси з `rag_eval_starter.json` переносяться в цю схему так:
`question` -> `input`, `expected_answer` -> `expected`.

**Трек C (ML)**: базовий потік `TRACK=C python src/generate.py` предиктить по рядках
`production_data.csv` і датасет не читає. Кейси в `eval_dataset.jsonl` для треку C — це твій
дизайн перевірок (зрізи, межові значення, пертурбації ознак); `input` — dict ознак, як рядок
`production_data.csv` без колонки `default`.
